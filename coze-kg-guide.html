STEP 8
上传 OpenAPI Schema
让 Coze 知道你的插件有哪些接口

在插件创建页面，找到「导入 API」→ 选择「从 YAML/JSON 导入」，粘贴 openapi_schema.yaml 的内容。

⚠️
重要：上传前把 schema 里的 url: https://your-deployed-domain.com 替换成你的真实 Railway 域名，否则 Coze 会请求错误的地址。

导入后 Coze 会自动解析出 5 个操作：

operationId方法路径用途
queryEntityPOST/query_entity精确/模糊查实体
queryNeighborsPOST/query_neighborsN 跳邻居查询
findPathPOST/find_path最短路径查找
fuzzySearchPOST/fuzzy_search实体消歧候选
healthCheckGET/health连通性检查

配置鉴权（API Key）

在插件的「鉴权」设置中：

字段填写
鉴权类型API Key

✅

STEP 12
逐节点配置详解
⚙ 全局 Workflow 变量（在开始节点配置）
user_query
类型 String — 用户输入的原始问题
hop_count
类型 Integer，默认值 0 — 当前已跳次数
max_hops
类型 Integer，默认值 3 — 最大允许跳数
context_graph
类型 String，默认值 "" — 累积的子图 JSON 字符串
current_entity
类型 String，默认值 "" — 当前要查询的实体名
found_answer
类型 Boolean，默认值 false
🧠 节点①：查询分析器（LLM Node）
模型
Claude-3.5-Sonnet 或 GPT-4o
输入变量
user_query
输出变量
start_entity（String）、query_intent（String）、sub_questions（String）
Prompt
你是知识图谱查询专家。分析用户问题，输出 JSON。 用户问题：{{user_query}} 要求输出纯 JSON，格式如下： { "start_entity": "从问题中提取的核心实体名（与图谱中一致）", "query_intent": "single_hop | multi_hop | path_finding | aggregation", "sub_questions": "将问题拆解为子问题的描述" }
⚡ 节点②：Cypher 生成器（Code Node）
语言
Python
输入变量
current_entity、hop_count、query_intent
输出变量
plugin_action（String）、plugin_params（String，JSON格式）
代码
PYTHON
import json

def main(current_entity: str, hop_count: int, query_intent: str) -> dict:
    # 根据意图选择调用哪个插件操作
    if query_intent == "path_finding":
        action = "queryNeighbors"
        params = {
            "entity_name": current_entity,
            "depth": 2,
            "limit": 15
        }
    elif hop_count == 0:
        # 第一跳：先查实体本身
        action = "queryEntity"
        params = {"entity_name": current_entity, "fuzzy": False}
    else:
        # 后续跳：查邻居
        action = "queryNeighbors"
        params = {
            "entity_name": current_entity,
            "depth": 1,
            "limit": 20
        }

    return {
        "plugin_action": action,
        "plugin_params": json.dumps(params, ensure_ascii=False)
    }
🔌 节点③：KG 查询执行（Plugin Node）
插件
选择你创建的「KG查询服务」
操作
使用 Code Node 输出的 plugin_action 动态决定（或固定选 queryNeighbors）
参数映射
把 plugin_params 里的 JSON 字段逐一映射到插件参数输入框
输出变量
kg_result（String，将插件返回的整个 JSON 转为字符串）
注意
Coze Plugin Node 不支持动态切换操作名，建议为每个插件操作各建一个 Plugin Node，通过前置 Branch 按 plugin_action 分流
🧠 节点④：跳转评估器（LLM Node）
输入变量
user_query、context_graph、kg_result、hop_count、max_hops
输出变量
should_continue（String: "true"/"false"）、next_entity（String）、updated_context（String）
Prompt
你是推理评估专家。判断是否需要继续查询。 原始问题：{{user_query}} 已查询跳数：{{hop_count}} / {{max_hops}} 已收集信息：{{context_graph}} 本次查询结果：{{kg_result}} 判断： 1. 当前信息是否已足够回答原始问题？ 2. 如果需要继续，下一步应该查哪个实体？ 输出纯 JSON： { "should_continue": "true 或 false", "next_entity": "下一跳要查询的实体名，不需要继续则为空字符串", "updated_context": "将本次 kg_result 合并到已有 context 的精简摘要", "reason": "判断理由（一句话）" }
🔀 节点⑤：条件分支（IF Node）
条件 A
should_continue == "true" 且 hop_count < max_hops → 连回「Cypher生成」节点，同时 hop_count + 1，current_entity = next_entity
条件 B（else）
→ 进入「答案合成」节点
变量更新
在分支 A 的连线上设置变量赋值：hop_count = hop_count + 1，current_entity = next_entity，context_graph = updated_context
✍️ 节点⑥：答案合成器（LLM Node）
输入变量
user_query、context_graph、hop_count
输出变量
final_answer（String）
Prompt
你是知识图谱问答助手。基于知识图谱查询结果回答用户问题。 用户问题：{{user_query}} 推理过程（共 {{hop_count}} 跳）：{{context_graph}} 要求： - 直接回答问题，语言自然流畅 - 在答案末尾附上「推理路径」，格式： 「推理路径：实体A → 关系X → 实体B → 关系Y → 实体C」 - 若信息不足，诚实说明并给出已知部分
STEP 13
连线规则说明
从到条件 / 备注
开始查询分析器直连，将 user_query 传入
查询分析器Cypher生成直连，初始时 current_entity = start_entity
Cypher生成KG查询直连
KG查询跳转评估直连
跳转评估条件分支直连
条件分支 TrueCypher生成（回环）⚠️ 这里形成循环，Coze 支持向前连线
条件分支 False答案合成直连
答案合成结束输出 final_answer

STEP 16
配置 System Prompt
# 角色设定 你是一个专业的知识图谱问答助手，基于结构化的教材知识图谱回答问题。 你拥有访问知识图谱数据库的能力，可以进行多跳推理。 # 能力说明 - 查询特定概念的定义和属性 - 探索概念之间的关系（支持最多 3 跳推理） - 发现两个概念之间的知识路径 - 归纳某类概念的所有相关实体 # 回答规范 1. 优先调用知识图谱工作流获取准确信息 2. 回答要准确、简洁，避免编造图谱中不存在的关系 3. 每次回答末尾附上「推理路径」，展示查询经过的节点 4. 若图谱中确实没有相关信息，诚实告知，不要猜测 5. 置信度不足时，说明"以下基于图谱中的有限信息" # 调用策略 - 问题包含实体名称 → 优先调用 多跳推理引擎 Workflow - 问题模糊或找不到实体 → 先调用 fuzzySearch 操作 - 问题涉及两个实体的关联 → 调用 findPath 操作 # 禁止行为 - 不得编造知识图谱中不存在的实体或关系 - 不得跳过工具调用直接凭记忆回答专业问题
STEP 17
绑定 Workflow 和 Plugin

在 Bot 编辑页面的「技能」区域：

技能类型添加内容说明
工作流多跳推理引擎主要推理入口，Bot 会自动决定何时调用
插件KG查询服务 → fuzzySearch单独暴露给 Bot，处理实体消歧
插件KG查询服务 → findPath单独暴露，处理路径发现问题
知识库（可选）上传教材 PDF 分段图谱查询失败时的语义兜底
💡
Workflow vs 直接调用 Plugin 的区别：多跳推理通过 Workflow 走，保证迭代逻辑正确执行。单次查询（如模糊搜索、路径查找）可以直接给 Bot 用插件，减少 Workflow 开销。
