PHASE 4 · WORKFLOW 多跳推理
STEP 11
新建 Workflow
coze.cn → 个人空间 → 工作流 → 创建工作流
字段填写
名称多跳推理引擎
描述接收自然语言问题，通过迭代查询 Neo4j 知识图谱实现多跳推理，返回结构化答案

Workflow 完整节点总览（最终实践版）：

FLOW主画布结构（循环节点实现多跳迭代）
开始(START)  输入参数：user_query
  ↓
① 查询分析器 [LLM]
  输出：start_entity、query_intent
  ↓
  变量赋值（进循环前，初始化三个全局变量）
  current_entity      = start_entity
  query_intent_global = query_intent
  user_query_global   = user_query
  ↓
┌─ 循环节点（无限循环）─────────────────────────────────────┐
│  [循环体内部]                                              │
│  ② Cypher生成器 [CODE]                                    │
│     读取全局：current_entity / hop_count                  │
│              / query_intent_global                        │
│     输出：action / entity_name / depth / limit / fuzzy    │
│     ↓                                                     │
│  ③ dispatch [PLUGIN]                                      │
│     ↓                                                     │
│  序列化 [CODE] → 变量赋值：kg_result_global = kg_result   │
│     ↓                                                     │
│  ④ 跳转评估器 [LLM]                                      │
│     读取全局：user_query_global / context_graph           │
│              / kg_result_global / hop_count / max_hops    │
│     输出：should_continue / next_entity / updated_context │
│     ↓                                                     │
│  选择器节点 [IF]                                          │
│  ├─ 继续 → ⑤-A 计算节点 [CODE]                          │
│  │          → 变量赋值：更新 hop_count / current_entity   │
│  │                      / context_graph（全局变量）       │
│  │          [自动进入下一次循环迭代]                       │
│  └─ 停止 → 终止循环节点                                  │
└────────────────────────────────────────────────────────────┘
  ↓ (循环结束后)
⑥ 答案合成器 [LLM]
  读取全局：user_query_global / context_graph / hop_count
  ↓
结束(END)
✅
两个关键设计决策（实践验证）：

① /dispatch 统一接口：后端新增此接口，Workflow 只用一个 Plugin Node，传 action 字段由服务端路由查询逻辑，彻底消灭分流 Branch。原方案「分流 Branch → Plugin A/B → 汇合」会触发 Coze「并发分支出线不允许进入另一个并发分支」限制。

② 循环节点（无限循环）：Coze 不支持在主画布直接绘制回环连线（报错「不允许成环」），必须使用内置「循环节点」实现迭代。循环体内可自由读写全局变量，循环节点本身无需配置中间变量，用「终止循环节点」控制退出。

逐节点配置详解
▶ 开始节点配置
输入参数

全局变量类型
选择 应用变量（每次新请求自动初始化为默认值，不选用户变量）

变量名类型默认值用途

🧠 节点①：查询分析器（LLM Node）
模型
Claude-3.5-Sonnet 或 GPT-4o
输入变量
user_query（引用开始节点输入参数）
输出变量
start_entity（String）、query_intent（String）、sub_questions（String）
Prompt
你是知识图谱查询专家。分析用户问题，输出纯 JSON，不要有任何其他文字。 用户问题：{{user_query}} 输出格式： { "start_entity": "从问题中提取的核心实体名（与图谱中一致）", "query_intent": "single_hop | multi_hop | path_finding | aggregation", "sub_questions": "将问题拆解为子问题的描述" }
📝 变量赋值节点（进循环前，初始化全局变量）
作用
循环体内部无法直接引用循环外节点的输出，必须先把关键值写入全局变量，循环体内再读取
目标全局变量值（引用节点①输出）
current_entity节点①输出 start_entity
query_intent_global节点①输出 query_intent
user_query_global开始节点输入参数 user_query
ℹ️
变量赋值节点只支持「变量引用」或「字面量」，不支持表达式运算（如 hop_count + 1）。需要运算时先用 Code Node 计算，再用变量赋值写回。

Code Node 书写规则：
① 函数定义必须用 async def；
② 所有参数必须给默认值（如 hop_count: int = 0）；
③ 类型注解冒号后必须有一个空格（如 str = ""，不能写 str=""）；
④ 不能输出 bool 类型——返回 True/False 会报 Output contain unknown value type，需改用 0/1 或直接在下游节点填字面量；
⑤ 循环体内读取应用变量（应用变量）可能被打包成 {"params":{...}} 传给第一个参数，需在函数开头用 json/ast.literal_eval 解包；对 Integer 类型应用变量还可能以 Args 对象传入（支持 == 比较，不支持算术），用 int() 转换或等值比较循环提取真实值；
⑥ Plugin Node 的嵌套对象数组输出无法可靠传递给 Code Node——Coze 对 schema 中 items: {type: object}（无具体 properties）的数组字段会丢弃内部数据，下游 Code Node 实际收到空数组。解决方案：后端在响应中额外附加一个 result_json 字符串字段（整个结果的 JSON 序列化），Code Node 只接收并透传该字段，彻底绕开嵌套对象传递问题。
🔁 循环节点配置
循环模式
无限循环（由循环体内的「终止循环节点」控制退出）
中间变量
不需要配置（状态完全通过全局变量传递）
ℹ️
Coze 不支持在主画布绘制回环连线（报错「不允许成环」）。必须使用循环节点实现迭代，不能直接拖线从下游节点连回上游节点。

── 以下为循环体内部节点 ──

⚡ 节点②：Cypher 生成器（Code Node，循环体内）
语言
Python
输入变量
全局变量 current_entity、hop_count、query_intent_global
输出变量
action（String）、entity_name（String）、depth（Integer）、limit（Integer）
⚠ 不输出 fuzzy —— Coze Code Node 不支持 bool 类型输出，fuzzy 在 dispatch 节点直接填字面量 false
代码
PYTHON
import json
import ast

async def main(current_entity: str = "", hop_count: int = 0, query_intent_global: str = "") -> dict:
    # Coze 循环体内读取应用变量时会打包成 {"params":{...}} 传给第一个参数，需解包
    p = None
    if isinstance(current_entity, dict):
        p = current_entity.get("params", current_entity)
    elif isinstance(current_entity, str) and current_entity.strip().startswith('{'):
        s = current_entity.strip()
        try:
            d = json.loads(s)
            if isinstance(d, dict):
                p = d.get("params", d)
        except Exception:
            pass
        if p is None:
            try:
                d = ast.literal_eval(s)
                if isinstance(d, dict):
                    p = d.get("params", d)
            except Exception:
                pass

    if p and isinstance(p, dict) and "current_entity" in p:
        current_entity      = str(p.get("current_entity", ""))
        hop_count           = int(p.get("hop_count", 0))
        query_intent_global = str(p.get("query_intent_global", ""))

    if query_intent_global == "path_finding":
        action = "queryNeighbors"
        depth = 2
        limit = 15
    elif hop_count == 0:
        # 第一跳：先查实体本身
        action = "queryEntity"
        depth = 1
        limit = 10
    else:
        # 后续跳：查邻居
        action = "queryNeighbors"
        depth = 1
        limit = 20

    return {
        "action": action,
        "entity_name": current_entity,
        "depth": depth,
        "limit": limit
    }

插件 / 操作
「KG查询服务」→ dispatch
参数映射
参数名来源

entity_name引用节点② 输出 entity_name

⚡ 序列化 Code Node（循环体内，紧接 dispatch 后）
输入变量
只引用 dispatch 输出的 result_json（String）字段——后端已将完整结果序列化为 JSON 字符串附在该字段，无需再接 entities/triples 等嵌套数组（Coze 无法可靠传递嵌套对象数组）
输出变量

代码
PYTHON
async def main(result_json: str = "{}") -> dict:
    return {"kg_result": result_json}
⚠️
为什么不直接接 entities/triples？Coze Plugin 对 OpenAPI schema 中 items: {type: object} 未定义属性的数组，传给 Code Node 时会丢弃内部数据（实测 entities 变为 []）。后端 /dispatch 接口在返回原始字段的同时，额外附加 result_json 字符串字段绕开此限制。参见后端 main.py dispatch 函数末尾：result["result_json"] = json.dumps(result, ensure_ascii=False)。
变量赋值
此节点后加变量赋值：kg_result_global = 本节点输出 kg_result
🧠 节点④：跳转评估器（LLM Node，循环体内）
输入变量
全局变量 user_query_global、context_graph、kg_result_global、hop_count、max_hops
输出变量
should_continue（String）、next_entity（String）、updated_context（String）
Prompt
你是推理评估专家。判断是否需要继续查询知识图谱。 原始问题：{{user_query_global}} 已查询跳数：{{hop_count}} / {{max_hops}} 已收集信息：{{context_graph}} 本次查询结果：{{kg_result_global}} 判断： 1. 当前信息是否已足够回答原始问题？ 2. 如果需要继续，下一步应该查哪个实体？ 输出纯 JSON： { "should_continue": "true 或 false", "next_entity": "下一跳要查询的实体名，不需要继续则为空字符串", "updated_context": "将本次查询结果合并到已有 context 的精简摘要", "reason": "判断理由（一句话）" }
🔀 选择器节点（IF，循环体内）
条件（继续）
should_continue == "true" 且 hop_count < max_hops → 走「计算节点」
否则（停止）
→ 走「终止循环节点」，退出循环
⚠️
比较值填写格式：右侧比较值填 true（纯字符串，不加引号）。若 Coze 自动加了转义引号变成 "\"true\""，条件将永远为 false，循环体直接走终止分支，多跳推理失效。
⚡ 节点⑤-A：计算节点（Code Node，选择器「继续」分支）
输入变量
全局变量 hop_count；节点④输出 next_entity、updated_context
输出变量
new_hop_count（Integer）、new_entity（String）、new_context（String）
代码
PYTHON
import json

async def main(hop_count=0, next_entity: str = "", updated_context: str = "") -> dict:
    # Coze 可能将所有输入（包括 LLM 输出）一起打包成 {"params":{...}} 传给第一个参数
    # hop_count 不加类型注解，以便接收 dict 或 int
    ah, an, ac = hop_count, next_entity, updated_context

    if isinstance(hop_count, dict):
        p = hop_count.get("params", hop_count)
        ah = p.get("hop_count", 0)
        if not an:
            an = str(p.get("next_entity", ""))
        if not ac:
            ac = str(p.get("updated_context", ""))

    count = 0
    try:
        count = int(ah)
    except (TypeError, ValueError):
        for i in range(10):
            if ah == i:
                count = i
                break

    return {
        "new_hop_count": count + 1,
        "new_entity": str(an),
        "new_context": str(ac)
    }

# 角色设定 你是一个专业的知识图谱问答助手，基于结构化的教材知识图谱回答问题。 你拥有访问知识图谱数据库的能力，可以进行多跳推理。 # 能力说明 - 查询特定概念的定义和属性 - 探索概念之间的关系（支持最多 3 跳推理） - 发现两个概念之间的知识路径 - 归纳某类概念的所有相关实体 # 回答规范 1. 优先调用知识图谱工作流获取准确信息 2. 回答要准确、简洁，避免编造图谱中不存在的关系 3. 每次回答末尾附上「推理路径」，展示查询经过的节点 4. 若图谱中确实没有相关信息，诚实告知，不要猜测 5. 置信度不足时，说明"以下基于图谱中的有限信息" # 调用策略 - 问题包含实体名称 → 优先调用 多跳推理引擎 Workflow - 问题模糊或找不到实体 → 先调用 fuzzySearch 操作 - 问题涉及两个实体的关联 → 调用 findPath 操作 # 禁止行为 - 不得编造知识图谱中不存在的实体或关系 - 不得跳过工具调用直接凭记忆回答专业问题

💡
Workflow vs 直接调用 Plugin 的区别：多跳推理通过 Workflow 走，保证迭代逻辑正确执行。单次查询（如模糊搜索、路径查找）可以直接给 Bot 用插件，减少 Workflow 开销。
